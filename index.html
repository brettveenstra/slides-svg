Icon Fonts

  • My first exposure was John Hicks' article for 24 ways
  • Where I learned the concept of embedding icons as characters in a font
  • But GitHub did a lot with their octicons to legitimize the technique
  • Icon fonts have proliferated
  • Pictos was one of the first
  • But others like Glyphicons
  • And especially Font Awesome really took off in popularity
  • Due in no small part to their inclusion in numerous popular frameworks
  • But like a lot of hacks, what may be brilliant at the time...
  • Tends to age rather poorly
  • I was emboldened to ditch icon fonts by Seren Davies' talk
  • Challenges for dyslexic users
  • Interfaces that relied on icon fonts to communicate important info
  • Would break when software was used to improve readability
  • Any part of the UI that relied on icon fonts is now unintelligible
  • I wrote about the issues Seren described in a blog post back in November
  • As well as several other issues
  • In certain situations, assistive devices may read certain characters aloud
  • Icon fonts can fail poorly, showing "missing character" blocks or even emojis
  • Your users could conceivable see horse emojis littering their star ratings...
  • Which is what happened to Etsy back in 2014
  • There are also many visual issues
  • They are monochromatic, a problem we've creatively attempted to sidestep
  • But thanks to font hinting built for letterforms
  • It never looked that great

Essentials

  • Here's the stuff you need to learn

Basic SVG Embeds 


<img src="foo.svg" alt="…">
          

.foo {
  background-image: url("foo.svg");
  /* etc. */
}

Acts like an image would (nothing fancy)

  • As mentioned earlier, you can embed SVGs just like any other image
  • But they act just like images
  • No interactivity, etc.

Interactive SVG Embed 


<object data="foo.svg" type="image/svg+xml">
  <!-- Fallback content -->
</object>

Interactive, accessible, cacheable, animatable, scriptable

  • You can embed an SVG with all its bells and whistles using the object tag
  • This way is nice because it's cacheable, has really straightforward fallback
  • The only thing it won't have is access to the containing page

SVG as HTML Element 


<p>
  There's an inline SVG right after this paragraph.
</p>

<svg>
  <!-- etc. -->
</svg>

Able to inherit in-page styles (currentColor and a lot more)

  • If you need the SVG to inherit page styles, etc.
  • You can include it directly in your page markup

SVG Container


<svg> </svg>
  • Regardless of where it lives, your SVG needs a container
  • That container may have some attributes

<svg xmlns="http://www.w3.org/2000/svg"
  xmlns:xlink="http://www.w3.org/1999/xlink">

Not necessary for <svg> elements in HTML5 pages

Otherwise, include them!

  • The XMLNS attribute tells some applications what types of XML tags it will be using.
  • Not necessary if included directly in HTML5 (implied by doctype)
  • Otherwise, be sure to include them!

<svg viewBox="0 0 100 100"
  width="100" height="100">
  • Dimension attributes
  • Width and height = "natural" dimensions (pixels)
  • Viewbox represents coordinates of image, regardless of display size
  • Leftmost, topmost, width and height

<svg viewBox="0 0 100 100"
  width="200" height="200">
  • If I double the width and height but leave the viewBox unchanged
  • Those 100 "points" of the SVG are stretched across the container

<svg viewBox="0 0 200 200"
  width="200" height="200">
  • It's only if I increase the size of the viewBox
  • So that more SVG points are shown
  • That the circle returns its original size

<svg viewBox="0 0 50 50"
  width="200" height="200">
  • If I decrease the width and height of the viewBox, it has the opposite effect
  • The 50-point viewBox stretches to fill the container, showing only the corner of the circle

<svg viewBox="-50 0 100 100"
  width="200" height="200">
  • Though you'll often use "zero" for the first two points, you don't have to
  • A negative value will start the visible content that many points to the left

<svg viewBox="0 0 100 100"
  width="100" height="200">
  • If the aspect ratio of the container differs from the viewBox
  • The image is scaled and centered, with more content visible above and below

<svg viewBox="0 0 100 100"
  width="100" height="200"
  preserveAspectRatio="none">
  • We can override this behavior with the preserveAspectRatio attribute
  • Setting it to "none" stretches the image

<svg viewBox="0 0 100 100"
  width="100" height="200"
  preserveAspectRatio="xMidYMin">
  • But we can also fix the image to a particular edge

<svg viewBox="0 0 100 100"
  width="100" height="200"
  preserveAspectRatio="xMidYMax">
          

<svg viewBox="0 0 100 100"
  width="100" height="200"
  preserveAspectRatio="xMidYMid slice">
  • Or even "slice" it to fill the entire container

});

Alternatives: Velocity.js, Snap.svg, KUTE.js, Animate Plus

  • Or JavaScript, which may be better for complex animations
  • Growing number of SVG-friendly animation libraries to choose from
  • This is just the beginning
  • SVG is its own document standard
  • Just as with HTML, every element has its own attributes, behavior, etc.
  • So what I just breezed through is really the tip of the iceberg
  • But if you learn even the basics of those elements and attributes...
  • You'll have everything you need to make the most of the format today

The Point

  • So here's my point
  • The web has a very specific cycle for new techniques and featuers

Lack of vector images

  • Our design intent encounters an obstacle

Icon fonts, Flash, crazy CSS shapes, etc.

  • We hack our way around that through any number of open or proprietary solutions

SVG!

  • Until a more robust, performant and predictable solution is standardized
  • SVG is that tool, and in a tremendous stroke of luck, it's actually better than what we thought we wanted (vector images)
  • "But SVG is hard" some say
  • No...

Change is hard
